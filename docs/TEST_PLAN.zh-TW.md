## 2. 簡介

### 2.1 目的

本文件定義 XRI Demo 專案的測試範圍、策略、環境、判定標準與測試案例,作為:

1. CI(持續整合)流程中自動化測試的正式依據;
2. 向利害關係人證明「大型 Windows C++ 檢測軟體可建立 CI 測試體系」的概念驗證文件;
3. 未來擴充測試時的格式與追溯範本。

### 2.2 受測系統概述

XRI Demo 模擬工業 X 光檢測軟體的核心工作流程:模擬 2D 掃描 → 結果影像檢視 → 檢測框(ROI)定義 → 演算法指派與檢測執行 → 結果匯出。架構分為無視窗相依的核心函式庫(`xri_core`)與 Qt Widgets 介面層(`xri_app_lib`)。

## 4. 測試範圍

### 4.1 納入範圍

- `xri_core` 核心函式庫全部模組:影像產生、掃描模擬、ROI 資料模型、清單模型、演算法引擎、報告匯出、工作階段狀態。
- `xri_app_lib` 介面層的關鍵操作流程:頁面導覽、掃描操作、滑鼠拖曳建立 ROI、演算法指派、檢測執行、按鈕啟用狀態。
- 跨平台建置與測試(MSVC / GCC)。
- Windows 發佈打包(windeployqt 產物的產生,由 CI Package 步驟驗證)。

### 4.2 不納入範圍(及理由)

| 排除項目 | 理由 |
| --- | --- |
| 視覺外觀比對(像素級截圖比對) | 概念驗證階段不需要;離屏截圖比對為未來可加入的測試層 |
| 真實硬體(偵測器、運動控制)整合 | 本專案以模擬器取代硬體;正式產品需另建硬體在環(HIL)測試站 |
| 效能 / 負載 / 長時間穩定性測試 | 非概念驗證目標 |
| 安裝程式(installer)測試 | 目前交付物為 zip;安裝程式屬未來工作 |
| 多語系 / 在地化測試 | 示範程式僅提供英文介面 |
| 檔案對話框(QFileDialog)互動 | 原生對話框會阻塞無螢幕測試;匯出邏輯改由 TC-EXP 系列直接驗證 |

## 5. 測試策略

### 5.1 測試層次

| 層次 | 套件 | 策略 |
| --- | --- | --- |
| L1 演算法層 | `tst_algorithms`、`tst_syntheticxray` | 以「答案已知的合成影像」(golden image)精確斷言量測值與判定,含門檻邊界測試 |
| L2 領域模型層 | `tst_inspectionplan`、`tst_scansimulator`、`tst_exporter` | 驗證資料模型契約、訊號行為、序列化往返、檔案輸出;以 `QAbstractItemModelTester` 自動檢查 Qt 模型契約 |
| L3 介面冒煙層 | `tst_smoke_mainwindow` | 以合成滑鼠/事件驅動真實 `MainWindow`,驗證端到端關鍵流程,不追求畫面細節覆蓋 |

### 5.2 測試設計原則

- **確定性優先**:影像產生使用固定種子與跨平台一致的亂數實作(自製 Box–Muller,避開 `std::normal_distribution` 的平台差異),使斷言可精確化。
- **零真實時間等待**:計時器間隔可注入為 0,掃描流程在事件迴圈內瞬間完成,避免 CI 上的不穩定(flaky)等待。
- **環境一致性**:`QT_QPA_PLATFORM=offscreen` 設定於 CTest 屬性,本機與 CI 執行 `ctest` 行為完全相同。
- **失敗安全預設**:未指派任何演算法時整體判定必須為 FAIL(不可因未檢測而誤判合格)。

### 5.3 執行方式

- 測試框架:Qt Test;測試執行器:CTest。
- 無 GUI 套件使用 `QTEST_GUILESS_MAIN`(僅 `QCoreApplication`);需要繪圖或視窗的套件使用 `QTEST_MAIN` + 離屏平台。
- 觸發時機:每次 `git push` 與每個 pull request 自動執行(雙平台);開發者本機可隨時以相同指令執行。

## 7. 進入與退出條件

### 7.1 進入條件(Entry Criteria)

- 原始碼可於目標平台完成 CMake 組態與建置,無建置錯誤。
- 6 個測試執行檔皆成功產出並註冊於 CTest。

### 7.2 退出條件(Exit Criteria)

- 全部測試案例 100% 通過(CTest 回報 0 failed)。
- 雙平台(Linux、Windows)皆通過。
- Windows 打包步驟成功產出 `xri_demo_win64.zip` 並上傳為 CI 產物。

### 7.3 暫停與恢復條件

- CI 任一測試失敗即視為該提交不可合併/不可發佈;修復並重新推送後自動恢復執行。

## 8. 判定標準

- **個別案例**:Qt Test 內所有 `QVERIFY` / `QCOMPARE` / `QTRY_*` 斷言成立即 PASS;任一斷言失敗、逾時或程序異常結束即 FAIL。
- **套件**:套件內全部案例 PASS 才視為 PASS。
- **整體**:6 個套件全數 PASS,且雙平台一致,才符合退出條件。

## 11. 測試排程與報告

- **觸發**:每次 push 與 pull request 自動執行(GitHub Actions);無人工排程。
- **報告**:CTest 輸出附於 CI 工作日誌;失敗時 `--output-on-failure` 自動列印完整斷言訊息;CI 狀態即測試報告之單一事實來源。
- **產物**:Windows 工作另產出 `xri_demo_win64.zip`(發佈驗收物)。
- **基準耗時**(快取後):Linux 全套件約 1 分 10 秒;Windows 含打包約 2 分 50 秒;本機全套件 < 5 秒。

## 12. 風險與限制

| 風險/限制 | 影響 | 緩解措施 |
| --- | --- | --- |
| 離屏測試不驗證視覺外觀 | 樣式/排版退化不會被攔截 | 發佈前人工目視檢查;未來可加入離屏截圖比對層 |
| FR-2(檢視器互動)覆蓋較薄 | 縮放/平移退化可能漏網 | 後續增加 ImageView 專屬單元測試 |
| 冒煙測試為單一線性流程 | 非主要路徑(取消、重掃、刪框後重跑)覆蓋有限 | 依風險逐步增加冒煙情境 |
| 真實硬體不在範圍 | 驅動層缺陷無法由本計畫攔截 | 正式產品建立硬體在環(HIL)夜間測試站 |
| 測試與文件可能脫節 | 本文件案例數/耗時隨程式演進過期 | 新增測試時同步更新本計畫(納入程式碼審查檢查項) |
